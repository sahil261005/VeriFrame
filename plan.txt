VeriFrame — Complete Implementation Plan
VeriFrame — Stateful Multi-Agent Video Deepfake Detection & Attribution Platform

A multi-agent AI system that detects deepfake videos using a pretrained ML model, classical computer vision, and LLM reasoning — orchestrated with LangGraph. Produces an explainable, confidence-scored verdict with frame-level attribution.

IMPORTANT

Coding Style Rule: All code must look like it was written by a competent junior developer. No over-engineered abstractions, no fancy metaclasses, no decorator factories, no abstract base classes with generics. Simple functions, simple classes, clear variable names, basic comments. See the 
Coding Style Guide
 section for details.

Coding Style Guide (Make It Look Human)
CAUTION

This is critical. If a recruiter or interviewer reads your code and thinks "this looks AI-generated," the project loses credibility. Follow these rules strictly.

DO (Junior Dev Patterns)
Simple functions that do one thing. No chaining, no clever one-liners.
Plain variable names: frames, score, result, frame_list. Not _frame_tensor_batch_normalized.
Basic for loops instead of nested list comprehensions.
Simple if/else blocks. Don't collapse everything into ternaries.
Comments that explain WHY, not WHAT. Write comments like a student would — explaining their thinking, not narrating every line.
Print statements for debugging left in non-critical paths (commented out is fine). Junior devs do this.
Simple try/except blocks with basic error messages. Don't write custom exception hierarchies.
Use requirements.txt, not pyproject.toml with poetry. Keep it simple.
Occasional slightly imperfect code — e.g., a variable declared a few lines before it's needed, a comment with a typo fixed later, a TODO for a minor improvement. Real code has rough edges.
DO NOT (AI Red Flags to Avoid)
❌ Abstract base classes or Protocol classes
❌ Custom decorator factories
❌ Type hints on every single variable (use them on function signatures only)
❌ Docstrings on every function in Google/NumPy format (use them sparingly, on complex functions only)
❌ Overly consistent formatting — real code has minor inconsistencies
❌ __all__ exports in every module
❌ Custom exception classes for every error type
❌ Generic type parameters (T, TypeVar)
❌ Perfectly alphabetized imports (group them logically but don't obsess)
❌ Enum classes for things that could just be strings
❌ Dataclasses with __post_init__ and validators for internal-only data
❌ Words like "leverage", "utilize", "orchestrate" in comments — use "use", "run", "call"
Example Comment Style
python

# BAD (AI-written):

# parameters and validates input constraints against system requirements.
# GOOD (human-written):

Phase 1 — Preprocessing Pipeline (Local)
Goal: Get video → frames + metadata working as standalone functions before touching any agent code.

File: backend/preprocessing.py
Functions to build:

validate_video(file_path)

Check file exists, check format (mp4, avi, mov, webm)
Check duration using OpenCV CAP_PROP_FRAME_COUNT / CAP_PROP_FPS
Reject if > 30 seconds
Return metadata dict: {fps, duration, width, height, codec, frame_count}
downscale_video(file_path, output_path, target_height=480)

Use OpenCV VideoCapture + VideoWriter
Read frame by frame, resize to 480p (maintain aspect ratio)
Write to a new temp file
Return path to downscaled file
extract_frames(file_path, interval=0.5)

Open video with OpenCV
Pull 1 frame every interval seconds (use frame index math, not timestamps)
Return list of dicts: [{frame_index, timestamp, image (numpy array)}]
Keep frames in memory (list of numpy arrays), do NOT write to disk
extract_audio(file_path, output_path)

Use subprocess to call ffmpeg for audio extraction
Extract audio track to a separate file (reserved for future use, not analyzed in v1)
If ffmpeg is not available or video has no audio, silently skip (return None)
get_video_metadata(file_path)

Wrapper that combines OpenCV props into a clean dict
{fps, duration, width, height, codec, total_frames, bitrate}
How to test Phase 1:
Download any 10-15 second video from YouTube (or record one on your phone)
Run preprocessing.py as a script with if __name__ == "__main__"
Print metadata, print number of frames extracted, show one frame with cv2.imshow()
Phase 2 — ML + CV Agents (Independent)
Goal: Get both analysis agents working as standalone functions before wiring them into LangGraph.

File: backend/agents/visual_agent.py
HuggingFace Model Selection:

Search HuggingFace Hub for "deepfake detection" or "deepfake image classification"
Pick a model that is:
Under ~200MB (ideally MobileNet or EfficientNet based)
Has a clear README with usage examples
Returns a binary classification (real vs fake) with confidence score
Good candidates to investigate at build time:
dima806/deepfake_vs_real_faces_detection (small, well-documented)
prithivMLmods/Deep-Fake-Detector-Model
Any ViT-based or MobileNet-based binary classifier
Functions to build:

load_model()

Use transformers.pipeline("image-classification", model="...")
Load once at startup, store in a module-level variable
Return the pipeline object
analyze_frames(frames, pipeline)

Loop through each frame (numpy array)
Convert to PIL Image (the transformers pipeline expects PIL)
Run inference, get label + confidence
Collect per-frame results: {timestamp, confidence, label}
Compute visual_score = average confidence across all frames that were labeled "fake"
Sort frames by "fake" confidence descending, pick top 5 as visual_flagged_frames
Return: visual_score, visual_flagged_frames
File: backend/agents/temporal_agent.py
Functions to build:

compute_optical_flow(frames)

Convert frames to grayscale
Use cv2.calcOpticalFlowFarneback() between consecutive frame pairs
Compute magnitude of flow vectors for each pair
Flag frame transitions where magnitude spikes abnormally (> 2 standard deviations from mean)
Return list of {timestamp, flow_magnitude, is_anomalous}
check_face_consistency(frames)

Use MediaPipe FaceMesh to detect facial landmarks on each frame
Track key landmark positions (nose tip, eye corners, chin) across consecutive frames
Compute Euclidean distance between corresponding landmarks in adjacent frames
Flag frames where landmarks jump by more than a threshold (sudden face shape changes)
Handle "no face detected" gracefully — if no face in most frames, return empty results with a note
Return list of {timestamp, landmark_shift, is_inconsistent}
run_temporal_analysis(frames)

Calls both functions above
Combines results: merges anomalous timestamps from both optical flow and face consistency
Compute temporal_score = proportion of frames flagged as inconsistent
Return: temporal_score, inconsistent_timestamps
How to test Phase 2:
Use the frames extracted in Phase 1
Run both agents independently on the same video
Print the scores and flagged timestamps
Test with a known deepfake video (download one from the DFDC sample set or FaceForensics++ preview clips)
Test with a known real video — scores should be low
Phase 3 — LangGraph Orchestration
Goal: Wire up the full multi-agent DAG with parallel execution, error handling, and the LLM reasoning agent.

File: backend/agents/state.py
Define VeriFrameState(TypedDict) with all fields from the spec
Keep it simple — just a TypedDict, no custom validators
File: backend/agents/llm_agent.py
Functions to build:

pick_suspicious_frames(visual_flagged, temporal_flagged, all_frames)

Merge the top flagged frames from both agents
Deduplicate by timestamp (if both agents flagged the same frame)
Cap at 8 frames max
Return list of frame dicts with their numpy arrays
analyze_with_gemini(suspicious_frames)

Use google.genai (Google AI Python SDK)
For each frame: convert numpy array to PIL Image, then to base64
Send to Gemini 2.5 Flash with a structured prompt asking for:
Lighting inconsistencies
Unnatural blending at face/hair boundaries
Shadow direction mismatches
Skin texture anomalies
Overall assessment (likely manipulated / likely authentic / uncertain)
Parse response into frame_explanations dict (keyed by timestamp)
Compile llm_reasoning (overall summary string)
Wrap in try/except — if Gemini fails (rate limit, timeout), return a default "LLM analysis unavailable" response, don't crash
Return: llm_reasoning, frame_explanations
File: backend/agents/synthesis_agent.py
Functions to build:

compute_verdict(visual_score, temporal_score, llm_reasoning, agent_status)

Default weights: visual=0.40, temporal=0.30, llm=0.30
Check agent_status — if any agent failed, redistribute its weight proportionally
e.g., if LLM failed: visual gets 0.40/(0.40+0.30) * 1.0 = 0.57, temporal gets 0.43
Compute weighted final_confidence
Determine verdict:
confidence > 0.70 → "MANIPULATED"
confidence < 0.30 → "AUTHENTIC"
else → "UNCERTAIN"
Return: final_verdict, confidence
build_report(state)

Compile a structured dict/JSON with:
Final verdict + confidence
Per-agent breakdown (individual scores, which frames each flagged)
LLM explanations per frame
Metadata (video duration, fps, total frames analyzed)
Agent health status (which agents succeeded/failed)
Whether this is a "partial analysis" (any agent failed)
Return the report dict
File: backend/agents/orchestrator.py
This is the LangGraph graph definition:

Define the graph nodes:

preprocess_node — already done by the time we enter the graph (frames passed in)
visual_node — calls visual_agent.analyze_frames()
temporal_node — calls temporal_agent.run_temporal_analysis()
llm_node — calls llm_agent.analyze_with_gemini() on merged flagged frames
synthesis_node — calls synthesis_agent.compute_verdict() + build_report()
Define the graph edges:


START → [visual_node, temporal_node] (PARALLEL)
[visual_node, temporal_node] → llm_node (SEQUENTIAL — waits for both)
llm_node → synthesis_node
synthesis_node → END
Error handling at each node:

Wrap each node function in try/except
If a node fails, set agent_status[agent_name] = "failed" in state
Set that agent's score to 0.0 and its flagged frames to empty list
Continue the graph — never halt
Compile the graph:

graph = StateGraph(VeriFrameState)
Add nodes, add edges, compile
Export a run_pipeline(frames, metadata) function that invokes the compiled graph
How to test Phase 3:

Test failure handling: temporarily break the Gemini API key — pipeline should still return a verdict with "partial analysis" flag

Interview Defense Prep
Key Questions to Prepare For:
Q: "Why multi-agent instead of just the ML model?"

"The baseline HuggingFace model achieved X% accuracy alone. By adding temporal consistency analysis (classical CV) and LLM reasoning, the full pipeline achieved Y% — a Z% improvement. The multi-agent design specifically reduced false positives because the temporal agent catches artifacts that the ML model misses, and the LLM provides explainable reasoning that a black-box classifier cannot."

Q: "Why LangGraph over just calling functions sequentially?"

"LangGraph gives me true parallel execution of the Visual and Temporal agents, typed state management so each agent reads/writes to a shared state dict without manual plumbing, and built-in failure handling with conditional routing. I could have written this as plain function calls, but LangGraph also gives me automatic tracing via LangSmith — I can see every agent's inputs, outputs, and timing in a dashboard, which was critical for debugging and tuning the pipeline."

Q: "Why weighted consensus instead of majority voting?"

"The three agents are not equally reliable. The HuggingFace model was trained specifically on deepfake datasets and carries more signal than optical flow heuristics. Weighted consensus lets me assign higher trust to the ML model (40%) while still incorporating the temporal (30%) and LLM (30%) signals. I tuned these weights based on evaluation results."

Q: "How do you handle Gemini rate limits?"

"The LLM agent only receives the top 5-8 most suspicious frames identified by the other two agents — never the full frame set. This keeps my Gemini calls to 5-8 per video analysis, well within the free tier's 15 requests/minute limit. If Gemini still fails, the pipeline gracefully degrades: the synthesis agent redistributes the LLM's 30% weight to the other two agents and flags the report as partial analysis."

Q: "What are the limitations?"

"Three main ones. First, heavy video compression (low bitrate YouTube rips) can mask manipulation artifacts, reducing both the ML model's and temporal agent's accuracy. Second, the system is primarily trained/tested on face-swap deepfakes — it may not detect other manipulation types like full-body puppetry or AI-generated scenes without faces. Third, adversarial perturbation attacks specifically designed to fool deepfake detectors would likely evade the ML model, though the temporal and LLM agents provide some resilience since they use fundamentally different detection approaches."

HuggingFace Model Selection: The exact model needs to be picked during Phase 2. This depends on what's available on HuggingFace Hub at build time and how it performs on your test videos. The plan above lists candidates, but the final choice will be made empirically.

IMPORTANT

Evaluation Dataset: DFDC sample set is the easiest to obtain (direct Kaggle download). FaceForensics++ requires an academic request form. Which one you use doesn't matter much for the resume — what matters is that you run the evaluation at all.

WARNING

Render vs HuggingFace Spaces: Render's 512MB RAM free tier will almost certainly not be enough for PyTorch + a deepfake model. Plan to use HuggingFace Spaces (Docker) for the backend from the start to avoid a painful migration later.
